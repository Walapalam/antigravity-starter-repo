# API Design Skill

## Description
This skill helps you design clean, intuitive, and maintainable APIs that provide a great developer experience.

## When to Use
- When creating new APIs or endpoints
- When designing function signatures
- When defining data structures and interfaces
- When planning system integrations
- When reviewing API designs

## How to Use
1. **Understand the use case**: Know what developers will do with the API
2. **Follow conventions**: Use established patterns in your ecosystem
3. **Design for clarity**: Make the API intuitive and self-documenting
4. **Plan for evolution**: Design with future changes in mind
5. **Consider errors**: Define clear error handling patterns
6. **Document thoroughly**: Provide comprehensive API documentation

## API Design Principles
- **Consistency**: Use consistent naming, patterns, and behaviors
- **Simplicity**: Make common tasks easy, complex tasks possible
- **Discoverability**: Make it easy to find and understand functionality
- **Flexibility**: Allow customization without over-complication
- **Stability**: Avoid breaking changes when possible
- **Performance**: Consider efficiency and scalability

## REST API Best Practices
- Use appropriate HTTP methods (GET, POST, PUT, DELETE, PATCH)
- Use meaningful resource names (plural nouns)
- Version your API (e.g., /api/v1/)
- Use standard status codes correctly
- Provide filtering, sorting, and pagination
- Return consistent response formats
- Use HTTPS for all endpoints
- Implement proper authentication and authorization

## Function/Method Design
- Use clear, descriptive names
- Keep parameter lists manageable
- Use sensible defaults
- Return consistent types
- Handle errors gracefully
- Avoid side effects when possible
- Make common cases simple

## Data Structure Design
- Use clear, meaningful field names
- Keep structures flat when possible
- Use appropriate data types
- Validate inputs
- Document constraints and requirements
- Consider backward compatibility

## Best Practices
- Write API documentation first (design-first approach)
- Get feedback from potential users early
- Provide code examples in documentation
- Use semantic versioning
- Deprecate gracefully with clear migration paths
- Test your API as a consumer would use it
- Monitor API usage and errors
- Provide helpful error messages

## Common Mistakes to Avoid
- Inconsistent naming conventions
- Over-complicating simple operations
- Insufficient error information
- Breaking changes without versioning
- Poor documentation
- Ignoring security concerns
- Not planning for scalability
