### Role of Workflows

Workflows within this architecture can serve as the "glue" that:

1. **Chain multiple skills together** in a specific sequence
2. **Define preconditions** (e.g., "plan must be approved before execution")
3. **Handle state management** (save artifacts to disk between steps)
4. **Enable complex behaviors** like parallel execution of independent tasks

### Example Workflow Structure

From the reference repository, here's how a workflow file looks:

```markdown
---
description: Executes an approved plan in small steps with verification after each step.
---

# Superpowers Execute Plan

## Preconditions (do not skip)

1. The user must have replied **APPROVED** to a written plan.
2. The approved plan must exist on disk at: `artifacts/superpowers/plan.md`

## Load the plan

- Read `artifacts/superpowers/plan.md`.
- Restate the plan briefly (1â€“2 lines) before making changes.

## Skills to apply as needed

Read and apply these skills when relevant:

- `superpowers-tdd` (preferred)
- `superpowers-debug` (if issues occur)
- `superpowers-review`
- `superpowers-finish`

## Execution rules (strict)

1. Implement **ONE** plan step at a time.
2. After each step:
   - Run verification command(s)
   - Append results to `artifacts/superpowers/execution.md`
3. If verification fails:
   - Stop and switch to systematic debugging
```

### Path Considerations

For workflows to reference skills correctly, we'd need to:

- Ensure correct relative paths from `.agent/workflows/` to `.agent/skills/`
- Document the expected directory structure in each workflow
- Provide helper scripts (like `write_artifact.py` in the reference repo) for common operations
